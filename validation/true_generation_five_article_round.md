目的：回应“是否真的实战生成”的问题。本轮不只做规则诊断，而是把 5 篇已有高质量文章压缩成真实用户可能给出的 `领域 / 困难` 输入卡，再按当前 `text-mining-intro-writing` skill 生成中文 omit-results Introduction，并用生成稿反推 skill 的更新要求。

## 1. S3LTWBGE：RAG 科学文献综合

### 领域 / 困难输入卡

| 项目 | 内容 |
|---|---|
| 领域 | 多学科科学文献综合问答；目标是回答专家提出的开放式科学问题。 |
| 困难 | 科学问题常需要跨多篇论文整合证据，长答案必须同时满足覆盖充分、引用准确、证据可追溯和专家可验证；通用 LLM 可生成流畅总结，但容易遗漏关键论文、错配引用、把无出处推断写成结论。 |
| 论文类型 | RAG / literature synthesis。 |
| 输出对象 | citation-grounded long-form answer / evidence-linked synthesis。 |
| 证据位置 | 检索段落、论文原文、引用编号、答案中的主张句。 |
| 错绑后果 | 引用编号支持不了对应主张，读者无法判断答案是基于证据还是模型补全。 |
| 方法方向 | 检索增强、证据段落绑定、自反馈修订、引用验证、覆盖检查。 |

### 实战生成 Introduction

科学文献正在成为科学推理本身的一部分，而不只是研究完成后的记录。许多前沿问题并不能由单篇论文直接回答：研究者需要比较不同实验条件下的发现，识别多个模型之间的共识与分歧，并把分散证据组织成可以被专家复核的长篇回答。这样的任务所需要的数据对象不是孤立的论文摘要，也不是若干关键词命中的段落，而是带引用支撑的综合性答案：每个关键判断都应能回到具体证据，每个证据又应在答案结构中承担清楚的作用[基准来源：S3LTWBGE；需补文献]。因此，科学文献综合的核心难题已经从“能否找到相关论文”转向“能否把多篇论文的证据正确地组织成可验证的论证”。

这一转向使传统检索和普通摘要之间的空隙变得突出。检索系统可以提高相关论文的可见性，却通常把判断负担留给读者；摘要模型可以压缩信息，却可能在没有充分证据的地方生成流畅但不可核查的概括。对开放式科学问题而言，真正危险的不是答案不够完整，而是答案看似完整却无法说明哪些句子由哪些论文支持。一个长答案若遗漏了相反证据，可能制造虚假的共识；若把引用编号放在不对应的主张后，可能把证据检索退化为形式化装饰；若只给出局部段落而缺少覆盖检查，专家仍需重新阅读大部分文献才能判断其可靠性。文献综合因此不是普通问答的放大版，而是覆盖、引用和论证结构同时受约束的证据组织任务。

已有路线分别缓解了这一任务的不同环节，但没有单独消除证据绑定瓶颈。人工综述能够保证专业判断，却更新慢、覆盖有限，并且难以复用为可交互问答系统。关键词检索和向量检索可以扩大召回范围，但它们返回的是候选证据，不负责判断证据是否足以支撑最终主张。通用大语言模型擅长把信息写成连贯段落，却容易把训练记忆、检索片段和推断性补全混合在一起。即使引入 RAG，如果生成阶段不检查引用是否真实支撑答案、不检查答案是否覆盖问题所需的主要方面，系统仍可能输出一种“有引用外观的幻觉”。真正未解决的问题，是如何让长答案中的每个主张、引用和检索证据形成可审计的三方关系。

在科学写作场景中，这种错配会直接改变答案的可信度。一个关于材料机制、生物通路或物理模型的问题，往往同时包含定义、实验条件、适用范围和争议边界；如果系统只检索到支持性论文，而没有覆盖反例或限制条件，答案会把局部结论写成一般规律。如果某个引用实际支持的是实验设置而非结论，读者会误以为结论已有文献证据。更隐蔽的是，模型可能把多篇论文中相邻但不兼容的发现拼接成一个更强的论断，从而把综合写成了未经验证的新假设。对专家而言，答案的价值不取决于语言是否顺滑，而取决于每个论断是否能沿引用回到可检查的证据段落。

因此，面向科学文献综合的 LLM 工作流应被设计为证据约束的生成系统，而不是自由摘要系统。更合适的路径是把检索、长答案组织、引用插入、证据段落回链、覆盖检查和自反馈修订放在同一个闭环中，使模型在生成每个主张时同时保留其证据来源，并在证据不足时标记不确定或拒绝过度概括。这样的系统需要把 citation-grounded answer 作为核心输出对象：答案不仅要读起来像综述，更要让专家能够追踪它如何选择论文、如何覆盖问题、如何把引用绑定到主张。只有在这一约束下，RAG 才能从“给 LLM 增加资料”转变为“让科学综合具有可审计证据结构”的方法。

### 可用性判断

可用等级：可用初稿，但不是投稿终稿。优点是没有被误写成材料数据库论文，RAG 类型、引用正确性、覆盖和专家验证都被放在核心位置。主要不足是领域不够具体，因为原文是多学科系统论文；若用于 Nature 投稿，需要补充 ScholarQABench、专家问题、评估方式等具体引用与背景，并把最后一段进一步压缩为更有力的研究空白。

## 2. UAL253B7：有机合成 ReactionSeek

### 领域 / 困难输入卡

| 项目 | 内容 |
|---|---|
| 领域 | 有机合成文献挖掘与反应知识发现。 |
| 困难 | 反应物、产物、用量、产率、温度、时间、SMILES/结构标签和语义反应信息分散在正文、反应式图像、表格和实验步骤中；同一化合物角色随上下文变化，图文对齐和条件-产率绑定困难，人工整理难以扩展。 |
| 论文类型 | Extraction / database。 |
| 输出对象 | evidence-linked reaction record。 |
| 证据位置 | 反应式图、实验步骤、底物范围表、补充信息、结构编号。 |
| 错绑后果 | 错把底物、产物、催化剂或条件绑定到相邻反应，导致条件-产率关系和反应适用性错误。 |
| 方法方向 | 领域 prompt、LLM、OCSR、化学归一化、证据回链。 |

### 实战生成 Introduction

有机合成文献的价值不只在于报告了新的分子或反应类型，更在于持续积累了可迁移的反应事实。对于路线设计、底物范围判断和反应条件推荐而言，真正有用的数据对象是 reaction record：它必须把底物、产物、试剂、催化剂、溶剂、温度、时间、产率、结构标识和原文证据绑定到同一个反应事件中[基准来源：UAL253B7；需补文献]。这些字段共同定义了“在何种条件下，何种反应物转化为何种产物，并得到何种结果”。如果只抽取化合物名称或数值，数据看似丰富，却无法支持可验证的反应规律学习。合成知识发现首先需要恢复这种反应对象，而不是简单扩大文本检索范围。

有机合成论文的写法恰好把 reaction record 拆散到多个位置。分子结构和编号常在反应式图或底物范围表中给出，实验步骤描述投料顺序、温度、时间和后处理，正文讨论代表性底物和失败情形，补充信息保存具体用量与表征证据。对化学家来说，图文对齐和编号追踪是阅读反应的基本动作；对自动抽取系统而言，这正是证据链最容易断裂的地方。一个产率可能对应表格中的某个底物编号，也可能对应正文中代表性实例；同一化合物在不同段落中可以是底物、产物、催化剂或添加剂。反应知识因此不是天然以机器可读记录存在，而是以跨图像、文本和表格的语义关系存在。

现有数据获取路线各有价值，却都容易在角色和条件绑定上留下缺口。人工数据库可提供高质量反应记录，但覆盖范围、更新速度和字段定制受到限制。OCSR 能从反应图像中恢复分子结构，却不保证结构编号已经与实验步骤和产率对齐。规则或传统 NLP 可以识别温度、时间、产率等文本字段，但面对多步反应、底物拓展表、缩写编号和不同期刊格式时容易失效。监督式关系抽取可以在特定语料上提高字段连接能力，却需要昂贵标注，且难以覆盖图像结构与实验文本之间的关系。LLM 具备上下文解释能力，但如果没有化学 schema、结构工具和证据定位，也可能把相邻反应混成一条看似完整的记录。

这种绑定错误会直接改变反应事实本身。一个化合物在主反应中是底物，在优化实验中可能是添加剂，在另一个例子中又可能成为产物；同一个温度可能属于反应阶段、退火处理、纯化步骤或表征条件；一个产率若被错配到底物范围表中的相邻编号，模型就会学到错误的底物适用性。尤其在多步合成中，中间体、目标产物和副产物的边界依赖上下文，字段齐全但角色错位的记录比缺失记录更危险，因为它会以“完整数据”的形式污染下游路线规划。对合成文献挖掘而言，核心问题不是能否识别化学名词，而是能否恢复 role-condition-yield-structure 的共同归属。

因此，更合适的路径是构建受化学 schema 约束、带证据回链的多模态 LLM 抽取流程。该流程应把 reaction record 作为唯一输出对象，要求每个字段同时给出结构编号、角色标签、条件归属、产率对应关系和证据位置；OCSR 与化学归一化工具用于约束分子结构，LLM 用于解释上下文和跨句连接，检索或证据回链用于把字段定位到反应式、表格或实验步骤。当结构编号、角色判断或条件归属证据不足时，系统应保留不确定性，而不是自动补全为一条完整反应。只有这样，文献中的合成事实才能转化为可审计、可更新、可定制的反应知识基础。

### 可用性判断

可用等级：较可用。生成稿已经抓住 ReactionSeek 的核心，即反应记录、角色变化、图文对齐和条件-产率绑定。与高质量原文相比，仍缺少更具体的有机合成知识发现动机，例如为什么 Organic Syntheses 这样的语料适合验证，以及 ReactionSeek 相比普通 LLM prompt 的独特定位。正式使用时应补充数据源、字段示例和一两个真实反应片段。

## 3. 58Z3MKQN：荧光粉发射波长预测

### 领域 / 困难输入卡

| 项目 | 内容 |
|---|---|
| 领域 | 发光材料 / Eu2+ 荧光粉发射波长预测。 |
| 困难 | 预测需要绑定材料配方、host-dopant 关系、激活离子价态、发射波长、测试条件和验证光谱；实体抽取只能找到公式和数值，不能保证发射峰属于正确主晶格、掺杂浓度和测试语境。 |
| 论文类型 | Relation prediction and validation。 |
| 输出对象 | host-dopant-emission relationship。 |
| 证据位置 | 正文、光谱图、图注、材料系列表、实验验证段。 |
| 错绑后果 | 错误学习主晶格-掺杂-发射关系，使波长预测和候选筛选失真。 |
| 方法方向 | 领域 NLP、化学感知解析、主体恢复、上下文消歧、ML 预测。 |

### 实战生成 Introduction

发光材料的机器学习设计依赖关系级数据，而不是孤立的发射峰数值。对于 Eu2+ 荧光粉，发射波长由主晶格局域环境、激活离子价态、掺杂浓度、晶场强度、激发条件和测试语境共同决定；同一个波长只有绑定到具体 host-dopant-emission relationship 中才具有可学习意义[基准来源：58Z3MKQN；需补文献]。因此，面向发射波长预测的文本挖掘任务并不是简单收集“材料名称 + 波长”，而是恢复“哪个主晶格中哪个激活离子在何种条件下产生何种发射”的关系对象。缺少这一关系，机器学习模型会把表面相似的光谱数值误当成可迁移规律。

相关信息大量存在于发光材料文献中，却被论文叙事自然拆分。材料配方和掺杂浓度可能出现在合成段或表格中，光谱峰位常位于图、图注或结果讨论中，激发波长、测试温度和样品系列编号又可能分散在实验方法和补充信息中。许多论文先定义一个主晶格系列，再逐步改变掺杂浓度或共掺杂元素，并在图中比较不同样品的发射峰位；如果抽取系统无法追踪“该样品”“最佳样品”或系列编号，发射波长就会脱离其主晶格和掺杂语境。对专家读者而言，这些关系可以通过上下文恢复；对自动系统而言，每一次省略和跨图引用都可能造成错误绑定。

已有材料文本挖掘方法可以回收部分字段，但难以保证关系语义完整。人工整理可以获得可信数据，却难以覆盖快速增长的荧光粉文献。通用化学解析器和 NER 能识别化学式、元素名和数值，却通常不判断数值是激发波长、发射峰、处理温度还是粒径。关系抽取模型可以连接材料和性质，但需要大量标注，并且面对省略主语、系列样品和图谱信息时容易失效。领域语言模型提高了术语识别能力，却未必能恢复 host、dopant、价态、浓度与 emission 的共同归属。真正限制下游预测的不是实体边界，而是关系是否代表正确的发光中心和测试条件。

错误绑定在荧光粉领域会被放大为错误的科学规律。相同的 Eu2+ 标记在不同主晶格和配位环境中可以产生完全不同的发射能级；相同的蓝光或红光峰位若没有激发波长、温度和主晶格约束，不能说明同一种结构-性能关系。如果抽取系统把图注中的发射峰分配给相邻样品，模型会错误学习掺杂浓度-波长趋势；如果把一个材料系列中的最佳样品性能误配给基准样品，候选筛选会偏向虚假的组合。更隐蔽的是，当论文没有明示价态或测试条件时，模型若按常识补全，会把推断写入训练数据。发射波长预测因此要求文本挖掘系统优先判断关系是否被证据支持。

因此，面向荧光粉的有效路线应把 NLP 抽取与关系约束的预测建模结合起来。抽取流程需要以 host-dopant-emission relationship 为核心 schema，同时保留配方、激活离子价态、浓度、激发/发射波长、测试条件和证据位置；化学感知解析用于规范化材料表达，主体恢复与上下文消歧用于处理系列样品和省略主语，图注和表格证据则用于校正光谱峰归属。当关系证据不足或字段来源冲突时，系统应标记缺失与不确定，而不是输出看似完整的材料-波长对。只有将关系级抽取作为预测数据的前置约束，文本挖掘才能真正服务于可验证的发光材料设计。

### 可用性判断

可用等级：可用。该稿较好继承了原文的 relation-prediction 逻辑，未把论文写成普通数据库构建，并且把 host-dopant-emission 关系放在开头。主要不足是 Nature 风格的开头还可以更强：应加入发光材料应用和高通量筛选需求，使“为什么需要预测”更有外部张力。

## 4. F9NLVIKW：材料系统图抽取

### 领域 / 困难输入卡

| 项目 | 内容 |
|---|---|
| 领域 | 冶金/合金材料系统图抽取。 |
| 困难 | 加工、结构、性能、机制以及信息来源标签分布在正文、图、表和讨论中；传统 NER/RE 容易把机制链压平成实体和值，无法保留 P-M-S-M-P 关系方向、机制与观察的区别、来源可追溯性。 |
| 论文类型 | System chart / knowledge graph。 |
| 输出对象 | P-M-S-M-P system chart / auditable mechanism chart。 |
| 证据位置 | 实验方法、显微图、性能表、讨论段、图注。 |
| 错绑后果 | 错误机制链会误导材料设计，把相关性写成因果，把外部常识写成原文证据。 |
| 方法方向 | LLM schema、节点/边类型、方向、来源标签、缺失关系标注。 |

### 实战生成 Introduction

冶金材料设计依赖机制链，而不是孤立的工艺名、组织术语或性能数值。加工路线如何改变组织，组织如何通过强化、相变、析出或缺陷机制影响性能，决定了合金设计原则能否迁移到新的体系。因而，面向冶金文献的关键数据对象应是 P-M-S-M-P system chart：它把 processing、mechanism、structure、mechanism 和 property 节点连接成带方向、带证据来源的关系图[基准来源：F9NLVIKW；需补文献]。如果这些关系被拆成普通字段，抽取得到的只是术语索引，而不是可用于设计推理的机制表示。

论文写作方式使这种系统图既存在又难以自动恢复。加工参数通常位于实验方法中，显微组织和析出相出现在图像、图注和表征段落中，力学性能保存在拉伸、硬度或冲击测试表里，机制解释则分布于讨论段甚至结论中。作者可以自然地把“时效处理导致析出相变化，进而通过沉淀强化提高强度”写成跨段叙事；抽取系统若只识别实体边界，就很难判断某个组织特征由哪一步加工导致，又通过哪种机制影响哪项性能。系统图的价值正在于把这种叙事压缩为可比较的关系对象，并保留每条边来自何处、方向为何、证据强弱如何。

已有自动化路线容易在这里丢失科学结构。人工阅读能够理解单篇论文的机制叙事，却无法把大量文献持续转化为统一、可检索、可绘图的系统图。NER 和表格抽取能找到“热处理”“析出相”“屈服强度”等实体或数值，却不负责区分 processing、structure、mechanism 和 property 之间的方向。知识图谱方法可以表达关系，但如果关系类型没有嵌入材料机制框架，就可能把共现误写成因果。LLM 可以生成流畅总结，却可能把原文证据、图像解释和外部常识混在一起。真正未解决的问题，是如何在自动抽取中同时保留机制方向、节点类型和信息来源标签。

系统图中的错误绑定会直接改变材料设计判断。相同的强度提升可能来自析出强化、细晶强化、固溶强化、相变诱导塑性或多机制耦合；相同的时效温度在不同合金体系中可能导致完全不同的析出路径。若系统把性能提升直接连到加工条件，而丢失中间组织和机制，读者就无法判断该工艺是否可迁移；若模型把外部常识推断出的机制标记为原文证据，系统图看似完整，却会误导后续知识综合。对冶金文献而言，缺失一条机制边和错误连接一条机制边同样重要，因为二者都会改变“为什么有效”的解释。

因此，更合适的文本挖掘路径是以 P-M-S-M-P schema 约束 LLM，把输出限制为可审计的 system chart。模型需要识别加工、机制、组织、机制和性能节点，并为每条边标注方向、来源位置和证据状态；当论文没有明确机制时，系统应保留缺失或标注推测来源，而不是自动补成完整因果链。该流程还应区分作者文本、图注、表格、模型推断与外部知识，使专家能快速定位薄弱关系。只有把系统图作为科学论证容器，而不是把它退化为数据库表，文献挖掘才能支持跨论文机制比较和材料设计推理。

### 可用性判断

可用等级：可用，但需补强领域细节。生成稿保留了系统图类型、边方向和来源标签，没有退化成数据库故事。弱点是冶金例子仍偏通用，最好补入具体合金体系、典型 P-M-S-M-P 链条和原文系统图定义，否则投稿时说服力不足。

## 5. PSZJCRRQ：材料设计假设生成

### 领域 / 困难输入卡

| 项目 | 内容 |
|---|---|
| 领域 | 低温 HEAs 与卤化物固态电解质的材料设计假设生成。 |
| 困难 | 目标不是抽取字段，而是生成超出设计者既有知识的、有协同机制、有科学依据、可被专家审计的设计假设；通用 LLM 可能提出流畅但空泛、无依据或不可实验检验的建议。 |
| 论文类型 | Hypothesis generation。 |
| 输出对象 | scientifically grounded design hypothesis / synergistic mechanism hypothesis。 |
| 证据位置 | materials system chart、机制链、CALPHAD 输入、加工-结构-性能信息、专家评价。 |
| 错绑后果 | 把无关机制拼接为看似新颖的假设，或把已有设计常识包装成“新假设”。 |
| 方法方向 | 从系统图生成假设，显式评估 synergy、scientific grounding、可审计性和专家可验证性。 |

### 实战生成 Introduction

材料设计中的智能化瓶颈正在从“能否收集知识”转向“能否提出有价值的设计假设”。对于低温高熵合金和卤化物固态电解质等复杂体系，研究者需要的不只是已有加工、结构和性能事实的整理，而是能够组合机制、识别协同效应并指向可验证实验的 hypothesis。这样的输出对象不同于数据库记录或文献摘要：一个有用的材料设计假设必须说明目标性能、可能的机制组合、适用材料空间、科学依据以及可被专家审计的证据链[基准来源：PSZJCRRQ；需补文献]。如果假设只是在语言上新颖，却不能说明为何可能成立，它就无法降低材料探索风险。

这一任务之所以困难，是因为“新颖”本身并不等于“有科学价值”。材料性能往往来自多个机制的协同或竞争，例如低温韧性可能同时受相稳定性、位错行为、晶界特征和元素分配影响；离子传导则可能依赖晶体框架、缺陷化学、界面稳定性和迁移能垒。文献和计算数据中确实包含这些线索，但它们通常以系统图、机制讨论、相图输入、性能表和专家解释的形式分散存在。设计者可以凭经验把这些线索组合成候选思路；自动系统若缺少机制约束，则容易把共现关系、相邻概念或常识性建议误写成“创新假设”。

已有路线各自解决了材料设计链条的一部分，却难以保证假设质量。人工专家能提出深刻假设，但受个人知识边界、阅读范围和时间成本限制。数据库筛选可以扩大候选空间，却通常依赖已有字段和既定指标，难以主动组合机制。知识图谱和系统图能够组织加工-结构-性能关系，但它们本身并不自动判断哪些关系组合具有协同潜力。通用 LLM 可以快速生成大量建议，却可能偏向高频材料常识，或者把缺少证据的推断包装成确定性结论。真正未解决的问题，是如何让假设生成同时满足新颖性、协同性、科学依据和可审计性。

错误假设比缺少假设更容易误导研究方向。若模型把两个并不兼容的机制简单拼接，实验可能沿着不存在的协同关系推进；若系统把已有设计常识改写为新提案，所谓“超出设计者知识”就只是语言重组；若假设没有标明来自哪些系统图节点、哪些 CALPHAD 输入或哪些实验事实，专家就无法判断其可信度和可验证路径。在复杂材料体系中，机制之间常常存在条件依赖：一种元素可能稳定目标相，也可能引入脆性相；一种缺陷可能提高离子迁移率，也可能降低化学稳定性。假设生成必须处理这种相互制约，而不能只追求流畅建议。

因此，面向材料设计的 LLM 工作流应以系统图和机制证据约束假设生成。更合适的输出不是开放式建议清单，而是带有目标、机制组合、协同理由、证据来源、不确定性和验证路径的 scientifically grounded design hypothesis。模型应从 materials system chart 中抽取可组合机制，并显式判断协同是否来自原文证据、计算输入、专家规则还是模型推断；当证据不足时，应标记假设的薄弱环节，而不是把推测写成结论。这样的流程还需要把专家评价或可审计标准纳入生成闭环，使假设质量不只由语言合理性决定，而由其科学 grounding 和实验可行动性决定。

### 可用性判断

可用等级：部分可用。该稿已经避免把 hypothesis-generation 写成普通抽取论文，并定义了 hypothesis、synergy、grounding 和 auditability。但与高质量原文相比，仍显得偏抽象：低温 HEAs 和卤化物固态电解质两个案例没有展开，导致机制协同的材料质感不足。若作为 Nature/Acta 风格投稿引言，需要加入更具体的领域痛点和案例级机制链。

## 本轮总体结论

1. 这次是真实生成：每篇都先有 `领域 / 困难` 输入卡，再由输入卡生成完整 omit-results Introduction。
2. 当前 skill 对 extraction、relation-prediction、system-chart 三类已经较稳；对 RAG 和 hypothesis-generation 能避免大方向错误，但生成内容仍容易偏抽象。
3. 主要失败不是“写不出来”，而是当输入卡只有困难摘要、缺少案例机制时，intro 会达到可用初稿水平，却难以达到高分文献的材料质感。
4. skill 需要新增一条强制要求：在 benchmark/迭代模式下，不能只做 gap analysis；必须执行 `原文证据 -> 领域/困难输入卡 -> 真实生成 -> 可用性判断 -> 规则更新`。
5. 对用户实际使用，输入中应强制包含：`领域`、`困难`、`输出对象`、`证据位置`、`错绑后果`、`方法方向`、`不可写的结果/贡献`。只给“文献很多、抽取困难”会导致泛化稿。
